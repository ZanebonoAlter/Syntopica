# OpenSpec 通用执行规范（Apply 阶段强制遵循）

&gt; 本文件为所有 Proposal 的默认施工纪律。  
&gt; 被 `@执行规范.md` 引用的 Proposal，必须在 Engineering Standards 章节完整并入以下内容。

---

## 1. 任务拆解规范
- apply 启动时，必须先执行 `/to-issues` 将 proposal 垂直切片为可独立交付的 GitHub Issues
- 每个 Issue 必须包含：验收标准、预估工时、关联的验收 checkbox

## 2. 实现纪律（TDD）
- **强制使用 `/tdd`**，严格遵循红绿重构循环：
  1. 只写 **1 个**失败测试
  2. 写 **最小**实现代码通过测试
  3. 重构优化，保持测试通过
- **禁止**：先写实现后补测试、一次生成多个测试、跳过重构阶段

## 3. 复杂设计验证
- 涉及以下场景时，**必须先 `/prototype` 做可丢弃原型**：
  - 状态机（≥3 个状态且存在循环流转）
  - 算法逻辑（排序、预测、匹配引擎）
  - 多模块交互协议（API 契约、事件总线）
- prototype 必须输出：方案对比、选型理由、风险点
- 确认后销毁原型，再进入 `/tdd` 正式实现

## 4. 质量门禁
- 全部子任务完成后，执行 `/setup-pre-commit` 配置：
  - Husky + lint-staged
  - Checkstyle / SpotBugs / ESLint（按技术栈）
- 提交前必须 100% 通过门禁，禁止 `--no-verify`

## 5. 架构体检
- 实现完成后，执行 `/zoom-out` 检查：
  - 新增模块与现有系统的耦合度
  - 是否引入循环依赖
  - 是否符合现有分层架构（如 Vert.x 的 EventBus 规范）
- 若耦合过高，必须输出 `/request-refactor-plan` 并经过 review

## 6. 变更控制（硬约束）
- **apply 阶段禁止修改 proposal.md 中的需求范围**
- 发现必须变更时，立即退出 apply，走 `opsx:delta` 或重新 propose
- 禁止在代码注释里写 "TODO：后续优化" 却不创建 Issue

## 7. 产出物清单（必须与代码一起交付）
- [ ] `ubiquitous-language.md`：本次变更涉及的新增领域术语
- [ ] 单元测试覆盖率报告（核心模块 ≥ 90%，整体 ≥ 80%）
- [ ] 性能基准测试报告（如涉及 DB 查询、并发处理）
