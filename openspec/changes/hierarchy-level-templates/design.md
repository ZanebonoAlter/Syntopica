## Context

当前标签层级系统通过 embedding 相似度和 LLM 判断建立父子关系，但缺乏对"抽象层级"的语义约束。这导致：

1. **跨层级错误**：具体事件挂在同类事件下（如 "OpenAI运营" 挂在 "DeepSeek融资" 下）
2. **层级过深**：Event 路径可达 7 层，每层都是模糊的事件描述
3. **类型混杂**：Keyword 是大杂烩，公司、技术、概念混在一起

现有系统通过 Phase 6 整树审查进行事后修复，但无法从根本上预防层级错误。

## Goals / Non-Goals

**Goals:**
- 为 Event、Person、Keyword 定义固定的层级模板，明确每层级的语义
- 标签创建时自动判定其抽象层级和细分类型
- 建立父子关系时强制校验层级约束
- 支持管理员调整层级定义，修改后生成待处理清单
- 为现有标签补充层级属性并标记违反新规则的关系

**Non-Goals:**
- 不引入新的 LLM 调用（复用现有 embedding 和判断流程）
- 不修改标签提取逻辑（只修改标签创建后的层级分配）
- 不支持按 Feed 或用户自定义模板（全局统一）
- 不自动修复违反规则的关系（仅标记待人工触发）

## Decisions

### 1. 固定模板，不可新增

**决策**: 系统预定义 5 个模板（event、person、keyword:tech、keyword:company、keyword:concept），不可新增，只能调整层级定义。

**理由**:
- 单用户系统，模板数量固定可管理
- 避免模板膨胀导致的维护复杂度
- 调整现有模板比新增模板更符合"打补丁"的维护模式

**替代方案**: 支持动态新增模板（拒绝：过度设计，单用户场景不需要）

### 2. 层级属性存储在 TopicTag 表

**决策**: `abstraction_level`（int）和 `sub_type`（string）作为 TopicTag 的新字段。

**理由**:
- 查询标签层级时无需 JOIN 配置表
- 层级是标签的固有属性，不应存储在外部配置
- 便于现有查询（如 `GetTagHierarchy`）直接过滤

**替代方案**: 存储在独立的 tag_metadata 表（拒绝：增加 JOIN 复杂度，层级是核心属性）

### 3. 待处理清单而非自动修复

**决策**: 模板修改后，扫描现有关系生成 `hierarchy_pending_changes` 记录，等待管理员手动触发 `rebuild`。

**理由**:
- 自动修复可能导致大量标签位置变动，影响用户体验
- 管理员需要审查每个变更的合理性
- 与现有"手动调整"的维护模式一致

**替代方案**: 自动修复 + 撤销功能（拒绝：实现复杂，单用户场景手动触发更可控）

### 4. SubType 判定：规则为主，LLM 兜底

**决策**: 通过规则库（关键词匹配）判定 keyword 的 sub_type，规则不确定时不调用 LLM，使用默认类型。

**理由**:
- 规则可覆盖 80% 的常见情况（含"公司"→company，含"框架"→tech）
- 减少 LLM 调用，降低成本和延迟
- 不确定时默认归类到 concept（最宽泛的 keyword 类型）

**替代方案**: 每次创建标签都调用 LLM 判定（拒绝：成本高，且 LLM 对"类型"的判断不稳定）

### 5. 配置存储为单条 JSON 记录

**决策**: `hierarchy_config` 表只有一行记录，`templates` 字段存储 JSON 配置。

**理由**:
- 模板结构简单（固定 5 个模板，每个 3-4 个层级），JSON 足够表达
- 单条记录避免复杂的关联查询
- 便于版本管理和整体回滚

**替代方案**: 关系型表结构（hierarchy_templates + hierarchy_levels）（拒绝：模板固定且层级少，关系型表增加复杂度）

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|---------|
| 现有标签层级属性 retroactive 填充工作量大 | 提供批量迁移脚本，在后台异步执行 |
| 规则判定 sub_type 可能有误判 | 管理员可在待处理清单中手动修正 |
| 模板调整导致大量待处理记录 | 添加批量忽略功能，支持按条件筛选 |
| 与现有清理调度器（Phase 6）冲突 | Phase 3.5 检查待处理清单，有记录时跳过自动审查 |
| 前端配置页面增加维护负担 | 配置页面极简（只改名称/描述/约束），无复杂交互 |

## Migration Plan

1. **数据库迁移**：
   - 新增 `hierarchy_config` 表（单条记录）
   - 新增 `hierarchy_pending_changes` 表
   - TopicTag 表增加 `abstraction_level`、`sub_type` 字段（默认 NULL）

2. **数据填充**：
   - 插入默认配置到 `hierarchy_config`
   - 异步为现有标签填充 `abstraction_level` 和 `sub_type`
   - 扫描现有关系，生成初始待处理清单

3. **代码部署**：
   - 部署新 API 和前端页面
   - 修改 `findOrCreateTag` 和抽象标签创建流程
   - 修改清理调度器

4. **验证**：
   - 检查新创建标签的层级是否正确
   - 检查待处理清单是否合理

5. **回滚**：
   - 配置回滚：更新 `hierarchy_config` 到上一版本
   - 代码回滚：不影响已有标签数据（字段可保留 NULL）

## Open Questions

1. **Event 类型的动态生成**: 是否需要 LLM 辅助生成新事件类型？还是纯规则+人工维护？
2. **Keyword sub_type 规则库**: 初始规则库需要多少条规则才能覆盖常见场景？
3. **层级深度限制**: Event 3层、Person 4层、Keyword 3层，是否预留弹性空间（如某些技术栈需要4层）？
4. **与现有 `tag-hierarchy-quality` 的集成**: 现有的 leaf-to-depth ratio 检查是否需要适配新模板？
