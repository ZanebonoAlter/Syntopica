# OpenSpec 通用执行规范（Apply 阶段强制遵循）

> 本文件为所有 Proposal 的默认施工纪律。
> 被 `@执行规范.md` 引用的 Proposal，必须在 Engineering Standards 章节完整并入以下内容。

---

## 0. 适用范围

本规范覆盖以下两套活跃代码栈：

| 栈   | 项目路径        | 构建工具 | 语言        |
| ---- | --------------- | -------- | ----------- |
| 后端 | `backend-go/`   | Go       | Go 1.25     |
| 前端 | `front/`        | pnpm     | TypeScript  |

额外测试套件：

| 套件         | 路径               | 框架     | 语言   |
| ------------ | ------------------ | -------- | ------ |
| 集成测试     | `tests/workflow/`  | pytest   | Python |
| Firecrawl 检查 | `tests/firecrawl/` | Python   | Python |

apply 启动时，根据变更所涉及的项目栈，对应执行相应章节的前置检查与质量门禁。

## 1. 任务拆解规范（通用）

- apply 启动时，将 proposal 垂直切片为 tasks.md 中可独立交付的子任务
- 每个子任务必须包含：验收标准、关联的验收 checkbox
- 若项目配置了 issue tracker，同步创建对应的 issue

## 2. 实现纪律 — TDD（通用）

> **实现前必须加载技能：** `tdd`

遵循铁律：**没有失败测试，不写产品代码。**

### 2.1 红绿重构循环

每次只做 **一个** 垂直切片：

```
RED   → 写 1 个测试，验证它失败（必须看到失败原因正确）
GREEN → 写最小实现代码通过测试
REFACTOR → 清理，保持所有测试通过
```

**严禁水平切片**（先写完所有测试再写实现）。每个测试必须在前一个测试 GREEN 之后才写。

### 2.2 强制规则

- 测试必须先于实现代码存在
- 每个测试必须亲眼看到失败，确认失败原因正确
- 实现代码必须是最小的 —— 只够通过当前测试
- 重构时保持所有测试绿色，不得添加新行为
- 测试描述行为而非实现，使用公共接口

### 2.3 分级执行

- **新增模块 / 核心逻辑变更** → **强制** 完整 TDD
- **Bug 修复** → **强制** 先写复现测试，再修代码
- **现有代码小幅修改** → 至少补充回归测试
- **纯配置 / SQL 迁移脚本** → 豁免（但需人工验证）

### 2.4 红旗行为 — 遇到即停止

- 先写实现后补测试
- 一次生成多个测试
- 测试立即通过（没有看到失败）
- 跳过重构阶段
- "这次先跳过 TDD"

遇到以上情况：删除已有代码，从 TDD 重新开始。

- 测试失败难以定位时，加载技能：`diagnose`

## 3. 复杂设计验证（通用）

> **进入原型阶段前必须加载技能：** `prototype`

- 涉及以下场景时，**必须先做可丢弃原型**：
  - 状态机（≥3 个状态且存在循环流转）
  - 算法逻辑（排序、预测、匹配引擎）
  - 多模块交互协议（API 契约、事件总线）
- prototype 必须输出：方案对比、选型理由、风险点
- 原型产出物归入 `openspec/changes/<name>/` 目录
- 确认后销毁原型，再进入 TDD 正式实现

## 4. 后端执行规范（Go + Gin + GORM）

### 4.0 前置条件检查

apply 启动时，先检测以下基础设施是否就绪，缺失则作为第一个子任务补齐：

- [ ] `go vet ./...` 可正常执行（0 错误）
- [ ] `golangci-lint run ./...` 可正常执行（已安装，配置文件 `.golangci.yml` 存在）
- [ ] `go test ./...` 可正常执行（即使 0 个测试用例）
- [ ] `go build ./...` 可正常编译

若 golangci-lint 未安装，作为首个子任务安装并配置：

```bash
go install github.com/golangci/golangci-lint/v2/cmd/golangci-lint@latest
```

### 4.1 质量门禁

提交前必须通过以下命令：

```bash
cd backend-go
golangci-lint run ./... && go vet ./... && go test ./... && go build ./...
```

门禁覆盖：

| 工具               | 用途                    |
| ------------------ | ----------------------- |
| `golangci-lint`    | 静态分析（staticcheck, revive, gosec 等） |
| `go vet`           | Go 官方静态检查          |
| `go test`          | 单元测试执行             |
| `go build`         | 编译检查                 |

提交前必须 100% 通过门禁。

### 4.2 测试规范

- 测试框架：标准 `testing` 包 + `github.com/stretchr/testify` 断言
- 测试文件以 `*_test.go` 与源码放在同一包中
- 需要数据库的测试：使用内存 SQLite（`glebarez/sqlite` + `mode=memory`），参考 `internal/domain/feeds/service_test.go` 的 `setupFeedsTestDB` 模式
- 优先使用表驱动测试（table-driven tests）
- setup 函数中使用 `t.Helper()` 获得更清晰的错误追踪

### 4.3 golangci-lint 配置要求

配置文件：`backend-go/.golangci.yml`

推荐启用的 linters：

| Linter          | 用途                        |
| --------------- | --------------------------- |
| `staticcheck`   | Go 静态分析（golint 的继任） |
| `go vet`        | 官方检查                    |
| `revive`        | 代码风格（golint 的替代品） |
| `gosec`         | 安全扫描                    |
| `errcheck`      | 未检查的错误返回值          |
| `gocritic`      | 代码质量建议                |
| `unused`        | 未使用的代码                |

### 4.4 后端代码规范

- 导入顺序：标准库 → 空行 → 第三方 → 空行 → 项目内部
- 命名：导出用 PascalCase，私有用 lowerCamelCase，包名简短
- 错误处理：`fmt.Errorf("context: %w", err)` 包装，禁止 `panic`
- JSON struct tags 使用 `snake_case`
- Handler 响应格式：`gin.H{"success": bool, "data"|"error"|"message": ...}`
- 业务逻辑在 `internal/domain/*`，不在 handler 或 router 中

## 5. 前端执行规范（Nuxt 4 + Vue 3）

> **前端实现前必须加载技能：** `vue-best-practices`

### 5.0 前置条件检查

apply 启动时，先检测以下基础设施是否就绪，缺失则作为第一个子任务补齐：

- [ ] ESLint 配置存在且可执行（`pnpm lint`）
- [ ] `pnpm exec nuxi typecheck` 可正常执行（0 错误）
- [ ] `pnpm test:unit` 可正常执行（使用 Vitest + happy-dom）
- [ ] `pnpm build` 可正常编译

若 ESLint 未配置，作为首个子任务安装并配置：

```bash
pnpm add -D eslint @eslint/js typescript-eslint
```

### 5.1 质量门禁

提交前必须通过以下命令：

```bash
cd front
pnpm lint && pnpm exec nuxi typecheck && pnpm test:unit && pnpm build
```

门禁覆盖：

| 工具                   | 用途                      |
| ---------------------- | ------------------------- |
| `pnpm lint`            | ESLint 代码检查            |
| `nuxi typecheck`       | TypeScript 类型检查        |
| `pnpm test:unit`       | Vitest 单元测试            |
| `pnpm build`           | Nuxt 生产构建              |

提交前必须 100% 通过门禁。

### 5.2 测试规范

- 测试框架：Vitest + Vue Test Utils + happy-dom
- 配置文件：`front/vitest.config.ts`
- 测试文件与源文件同目录，命名 `*.test.ts`（如 `app/utils/foo.test.ts`）
- E2E 测试使用 Playwright，放在 `front/tests/e2e/*.spec.ts`
- 使用 `describe`/`it` 块，测试描述行为而非实现

### 5.3 ESLint 配置要求

配置文件：`front/eslint.config.js`（flat config）

推荐规则：

| 规则类别     | 说明                                    |
| ------------ | --------------------------------------- |
| JS 基础      | `@eslint/js` recommended                |
| TypeScript   | `typescript-eslint` strict-ish          |
| Vue          | `eslint-plugin-vue` recommended         |

### 5.4 前端代码规范

- 组件使用 `<script setup lang="ts">`（Composition API）
- 组件名 PascalCase，composable 名 camelCase + `use` 前缀
- 导入顺序：Vue/Nuxt → 第三方 → 内部 → type-only
- 使用 `import type` 导入纯类型
- API 调用通过 `app/api/client.ts` 的 `ApiClient`，不在组件中直接调用
- `snake_case → camelCase` 转换在 store/API 层完成，模板中不出现 snake_case
- 数字 ID 在 API 边界转字符串

## 6. 集成测试规范（Python pytest）

### 6.0 前置条件

- [ ] Python 环境已配置（`tests/workflow/.venv` 存在）
- [ ] `tests/workflow/requirements.txt` 依赖已安装
- [ ] Go 后端运行在 `http://localhost:5000`

### 6.1 运行方式

```bash
cd tests/workflow
.venv\Scripts\activate     # Windows
pytest test_*.py -v
```

### 6.2 编写规范

- 测试文件放在 `tests/workflow/test_*.py`
- 使用共享辅助工具：`utils/database.py`（DatabaseHelper）、`utils/api_client.py`（APIClient）、`utils/mock_services.py`（MockFirecrawl / MockAIService）
- 每个 test class 使用 `pytest.fixture(autouse=True)` 进行 setup/teardown
- 测试配置见 `tests/workflow/config.py`

## 7. 架构体检（通用）

实现完成后，执行以下检查：

### 7.1 GitNexus 知识图谱检查（强制）

- 每个子任务完成后必须执行：
  1. `gitnexus_impact({target: "修改的符号", direction: "upstream"})`
     → 确认无 HIGH/CRITICAL 风险被忽略
  2. `gitnexus_detect_changes({scope: "all"})`
     → 确认变更范围符合预期
- 若 impact 返回 HIGH/CRITICAL 风险，必须暂停并向用户报告，不得忽略

### 7.2 架构合理性检查

- 加载技能 `zoom-out` 检查：
  - 新增模块与现有系统的耦合度
  - 是否引入循环依赖
  - 是否符合现有分层架构（后端：`internal/domain/*` + `internal/platform/*`；前端：Vue 3 Composition API + Pinia stores）
- 若耦合过高，暂停并提出重构建议

## 8. 变更控制（硬约束）

- **apply 阶段禁止修改 proposal.md 中的需求范围**
- 发现必须变更时，立即退出 apply，走 `opsx:delta` 或重新 propose
- 禁止在代码注释里写 "TODO：后续优化" 却不记录到 tasks.md

## 10. 数据兼容性检查

涉及以下后端数据层变更时，必须在 tasks.md 中增加兼容性验证子任务：

- **PostgreSQL DDL 变更**（GORM AutoMigrate / 手动迁移）→ 确认既有数据兼容，确认查询不受影响
- **数据库迁移脚本**（`cmd/migrate-db/`）→ 确认迁移可重复执行（幂等），确认回滚路径
- **GORM 模型字段变更** → 确认 JSON API 响应格式不受破坏性影响
- **配置文件**（`config.yaml`）新增或修改字段 → 同步更新 `docs/reference/configuration.md`
